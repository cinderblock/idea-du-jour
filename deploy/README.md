# Deploying idea-du-jour to firefly

idj runs on `firefly` as an **ops-managed Docker "stack"** (like `nginx-cache`):
CI builds an image and pushes it to ghcr; firefly's `deploy.sh` does
`docker compose pull` + `up -d` for the stack; Caddy reverse-proxies
`https://idj.isozilla.com` → the container's loopback port.

```
push idj ─▶ GitHub Actions (build.yml) ─▶ ghcr.io/<owner>/idea-du-jour:latest
                                                    │
ops deploy (cron / ops push / manual) ─▶ deploy.sh ─┤ docker compose pull + up
                                                    ▼
                                   idj container  127.0.0.1:18787:3000  ── data volume (/data)
                                                    ▲
Cloudflare (idj.isozilla.com, proxied) ─▶ Caddy ───┘  reverse_proxy 127.0.0.1:18787
```

## App-side (this repo — done)

- `Dockerfile` — Bun build + slim runtime; runs migrations on boot, serves the
  Nitro output. Listens on `PORT` (3000), `DATABASE_URL=file:/data/idj.db`.
- `.github/workflows/build.yml` — builds and pushes `ghcr.io/<owner>/idea-du-jour`
  (`:latest` + `:sha`) on push to main/master.

## Prerequisites (need you)

1. **Put idj on GitHub** (e.g. `cinderblock/idea-du-jour`). The image name in the
   stack compose assumes owner `cinderblock` — adjust if different.
2. **Make the ghcr package public.** firefly's stack `docker compose pull` runs
   *without* a registry login (login only happens later, for Caddy), so the image
   must be public. (Alternative: move `docker login` before the stacks loop in
   `deploy.sh` — a separate ops change.) After the first CI run, set the package
   visibility to Public in the repo's Packages settings.
3. **Session secret** is auto-generated by the stack (`env.json`) — nothing to set.
4. **Enrichment is off in prod** (no `ANTHROPIC_API_KEY`). The triage skill (Max
   plan) is the primary processing path. To turn on auto-enrich later, add
   `ANTHROPIC_API_KEY` to `stacks/idj/env.json` + set `FIREFLY_ANTHROPIC_API_KEY`.

## Ops-side changes (GATED — staged in `deploy/ops-staged/`, apply with authorization)

Each is a per-change infra edit; apply through the ops repo after approval:

1. **DNS** — `deploy/ops-staged/cloudflare-dns.md` (one line in
   `cloudflare/config/isozilla/isozilla.com.yaml`). Validate with `bun run sync`.
2. **Caddy site** — copy `deploy/ops-staged/firefly/sites.d/idj.isozilla.com.caddy`
   to `servers/firefly/sites.d/` in ops.
3. **Stack** — copy `deploy/ops-staged/firefly/stacks/idj/` to
   `servers/firefly/stacks/idj/` in ops.

Committing 2 + 3 to ops `master` triggers the ops deploy workflow (it runs on
`servers/**` changes), which brings up the stack and hot-reloads Caddy.

## Go-live order

1. idj → GitHub; first CI build succeeds; make the ghcr package public.
2. Apply ops DNS + Caddy + stack (gated). Push ops → deploy runs.
3. Visit `https://idj.isozilla.com` → redirected to `/login` → create your passkey.
4. Update the triage skill's `.env.local`: `IDJ_BASE_URL=https://idj.isozilla.com`
   and a fresh agent token minted against prod.

## Updating after go-live

Push idj → CI republishes `:latest`. firefly picks it up on its next ops deploy
(weekly cron, any ops push, or `gh workflow run deploy.yml -R <owner>/ops`). To
auto-trigger on idj push, wire the optional dispatch step in `build.yml`.
